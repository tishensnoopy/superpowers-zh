# 项目开发规则

## 1. 项目概述

本项目是基于 Strapi v5 CMS 和 React 18 的企业官网，采用前后端分离架构：

| 层级 | 技术栈 | 说明 |
|------|--------|------|
| 前端 | React 18 + TypeScript + Vite | 页面渲染、组件交互 |
| 路由 | react-router-dom v6 | 单页应用路由管理 |
| 后端 | Strapi v5 | CMS 内容管理、API 提供 |
| 测试 | vitest + @testing-library/react | 单元测试、集成测试 |

## 2. 开发流程规范

### 2.1 TDD 流程

严格遵循测试驱动开发流程：

```
RED → GREEN → REFACTOR
```

1. **RED**: 先编写失败的测试用例
2. **GREEN**: 实现代码使测试通过
3. **REFACTOR**: 优化代码，保持测试通过

### 2.2 代码审查

- 每个功能完成后必须进行代码审查
- 审查重点：逻辑正确性、代码质量、测试覆盖率
- 使用 `requesting-code-review` 技能进行审查

### 2.3 验证完成

- 功能实现完成后必须运行验证命令
- 使用 `verification-before-completion` 技能
- 所有测试通过后才能提交

## 3. 测试规范（核心）

### 3.1 数据格式覆盖原则

**规则**: 涉及外部 API 数据的组件测试，必须同时覆盖文档格式和实际 API 返回格式

**示例**:
- Strapi v4 格式: `{ data: [...] }`
- Strapi v5 格式: `[...]`（直接数组）

**测试要求**: 每个涉及 API 数据的组件必须同时测试 v4 和 v5 两种格式

**额外边界情况**:
- API 字段缺失（后端字段被移除）
- API 字段类型变化（string → number）
- 嵌套结构变化（attributes 嵌套 → 扁平）
- API 版本升级导致的数据格式变更

### 3.2 边界情况测试

**规则**: null、undefined、空数组、空对象等边界场景必须覆盖

**测试清单**:
- `null` 值处理
- `undefined` 值处理
- 空数组 `[]`
- 空对象 `{}`
- 空字符串 `''`
- 异常数据类型（如数字传入字符串字段）
- 极端值（超长字符串、特殊字符）
- 异步操作超时
- 并发请求（如重复提交）
- 时区处理（日期时间显示）
- 网络异常（断网、慢网）

### 3.3 浏览器视觉验证

**规则**: 验证检查点必须使用真实浏览器验证，不能只依赖单元测试

**验证内容**:
- 页面渲染完整性
- 组件布局正确性
- 交互功能正常
- 控制台无 error/warning

### 3.4 响应式测试

**规则**: 必须覆盖桌面(1280px)、平板(768px)、手机(375px)三种宽度

**测试要点**:
- 断点边界值测试（如 768px、375px 的精确值）
- 横向滚动问题检查
- 固定定位元素的响应式表现
- 图片和媒体的自适应

### 3.5 路由上下文原则

**规则**: 使用 React Router hooks 的组件测试，必须提供 Router 上下文

**示例**:
```typescript
// 测试中必须使用 MemoryRouter 包裹
render(
  <MemoryRouter>
    <Routes>
      <Route path="/" element={<ContactForm section={section} />} />
    </Routes>
  </MemoryRouter>
);
```

### 3.6 路由安全原则

**规则**: 需要通过路由 state 传递数据的页面，必须验证数据有效性

**边界情况覆盖**:

| 场景 | 处理方式 | 测试要求 |
|------|---------|---------|
| 用户直接访问成功页 | 显示"访问受限"页面 | ✅ 必须测试 |
| 表单提交后刷新页面 | 使用 localStorage 持久化数据（可选） | ✅ 建议测试 |
| 用户篡改 state 数据 | 验证关键字段格式 | ✅ 必须测试 |
| 用户通过后退按钮返回 | 重新验证数据有效性 | ✅ 必须测试 |
| state 数据不完整 | 显示友好错误提示 | ✅ 必须测试 |
| state 数据类型错误 | 类型检查和容错处理 | ✅ 建议测试 |
| 多标签页/窗口状态隔离 | 使用唯一 token 标识会话 | ✅ 建议测试 |
| 超时过期 | 设置过期时间，超时后重新验证 | ✅ 建议测试 |
| 浏览器后退后再前进 | 重新验证数据有效性 | ✅ 必须测试 |
| 多页面流程状态丢失 | 使用全局状态管理或 URL 参数 | ✅ 建议测试 |

**实现示例**:
```typescript
useEffect(() => {
  if (!location.state?.appointment?.phone) {
    // 验证失败，显示访问受限页面或重定向
  }
}, [location.state]);
```

**验证标准**:
- 以必填字段（如 phone）作为验证依据
- 验证失败时显示友好的访问受限页面
- 引导用户返回正确流程
- 测试必须覆盖"有数据"和"无数据"两种场景

## 4. 代码规范

### 4.1 TypeScript 规范

- 使用严格模式（`strict: true`）
- 避免 `any` 类型，使用明确的类型定义
- 组件 props 必须定义接口
- 函数参数和返回值必须标注类型

### 4.2 React 规范

- 使用函数组件和 Hooks
- 避免使用类组件
- Hooks 必须遵循依赖数组规则
- 组件命名使用 PascalCase

### 4.3 代码风格

- 使用 ESLint 进行代码检查
- 使用 Prettier 进行格式化
- 缩进使用 2 个空格
- 文件末尾保留空行

## 6. 经验教训记录

### 6.1 测试程序不完整的教训

**问题**: 初版测试只覆盖 `{data: [...]}` 格式（Strapi v4），未覆盖 Strapi v5 实际返回的直接数组格式，导致"测试通过但生产环境失败"。

**根本原因**: 测试 mock 数据格式与实际 API 返回格式不一致。

**影响范围**: Advantages、Team、ContactForm 三个组件。

**修复方案**: 补充 Strapi v5 直接数组格式的测试用例。

### 6.2 useNavigate 上下文问题

**问题**: ContactForm 组件使用 `useNavigate()` 后，原有测试因缺少 Router 上下文而失败。

**根本原因**: React Router hooks 必须在 Router 组件内部调用。

**修复方案**: 在测试中使用 `MemoryRouter` 包裹组件。

### 6.3 路由安全验证缺失（重大问题）

**问题**: 预约成功页没有验证是否真的有预约数据，任何人都可以直接访问 `/appointment-success`，导致显示"预约成功"但数据为空（显示"未填写"），容易造成用户误解和信任危机。

**根本原因**:
1. 成功页没有验证 `location.state` 是否存在
2. 没有检查关键数据字段（如 phone）是否有效
3. 缺乏路由安全防护机制

**修复方案**:
1. 添加 `useEffect` 验证 `location.state?.appointment?.phone` 是否存在
2. 验证失败时显示"访问受限"页面，引导用户返回首页预约
3. 添加 loading 状态，避免闪烁

**边界情况补充**:
- 页面刷新导致 state 丢失：考虑使用 localStorage 持久化
- 用户篡改 state：验证关键字段格式和长度
- 后退按钮返回：重新验证数据有效性
- 数据类型错误：添加类型检查和容错

**影响**:
- 测试用例从 5 个增加到 9 个（有数据场景 + 无数据场景）
- 测试总数从 39 个增加到 43 个

## 7. 安全规范

### 7.1 前端安全

- 输入验证（前端 + 后端双重验证）
- 防止 XSS 攻击（使用 React 自动转义）
- 敏感数据不存储在 localStorage
- API 请求使用 HTTPS
- CSRF 防护（使用 anti-CSRF token）
- 避免使用 `dangerouslySetInnerHTML`，必须使用时进行内容过滤

### 7.2 后端安全

- 频率限制（每个 IP 每小时最多提交 5 次）
- 记录 IP 地址和 User-Agent
- 路由配置限制不必要的操作
- 预约数据仅管理员可见
- CORS 配置（限制允许的源）
- 日志脱敏（不记录敏感信息如手机号）
- SQL 注入防护（使用参数化查询）

## 8. 性能规范

### 8.1 代码优化

- 使用 React.memo 避免不必要的重渲染
- 使用 useCallback 缓存回调函数
- 图片懒加载
- 代码分割和按需加载
- 首屏加载时间控制在 2 秒以内
- 大型列表使用虚拟滚动或分页

### 8.2 构建优化

- 启用 Vite 压缩
- Tree Shaking 去除未使用代码
- CDN 加速静态资源
- 缓存策略配置
- 移除未使用的依赖
- 生产环境关闭 source map

## 9. 团队协作规范

### 9.1 Git 规范

- 使用 Conventional Commits 格式
- 分支命名规范: `feature/<feature-name>`
- 每个 PR 解决一个问题
- 提交前运行测试

### 9.2 代码审查

- 每个 PR 必须至少 1 人审查
- 审查重点: 逻辑正确性、代码质量、测试覆盖率
- 审查意见必须逐条回应

### 9.3 文档维护

- 代码变更后同步更新文档
- 文档保持与代码一致
- 删除过时文档

## 10. 故障排查规范

### 10.1 问题定位

- 先查看控制台日志
- 检查 API 响应数据
- 验证数据格式
- 逐步缩小范围

### 10.2 修复流程

- 重现问题
- 分析根本原因
- 编写测试用例
- 实施修复
- 验证修复

### 10.3 预防措施

- 添加针对性测试用例
- 更新相关文档
- 纳入项目规则
